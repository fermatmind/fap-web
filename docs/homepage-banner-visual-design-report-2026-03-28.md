## 一、总目标

首页前四屏必须形成一条完整的“系统运行链路”。

### Banner 1：系统启动舱

识微 · 见远，启动“决策级自我认知引擎”。

### Banner 2：校准实验台

选择正确的判断入口，而不是选一个娱乐测试。

### Banner 3：验证机架 / 计算内核

展示费马测试为什么不是标签游戏，而是有方法、有常模、有映射逻辑的系统。

### Banner 4：证据沉降 / 审计档案区

把高压系统感收束为可信的历史验证记录与社会证明。

### 用户最终感受

用户不应该觉得自己在浏览几个独立模块，而应该明显感到：

> 我不是在浏览几个模块，我是在进入一台正在运行的高精度认知系统。

## 二、总体设计原则

### 1. 视觉路线不换赛道

必须保留：

- 冷静
- 精密
- 工业系统感
- 数据可信感
- 非情绪化表达
- 非心理测试娱乐风

坚决不要：

- 普通 SaaS 白板风
- 温柔治愈插画风
- 渐变紫 AI 模板风
- 大圆角卡片商城风
- 大面积玻璃拟态漂浮感
- 花哨营销页装饰语言

### 2. 明度节奏

首页中段的明度节奏必须是：

深 → 浅 → 深 → 浅收束

也就是：

- Banner 2：浅色实验台
- Banner 3：深色验证主场
- Banner 4：浅色审计档案区

但这个“浅 / 深 / 浅”不是切页效果，而是有语义理由的系统级过渡。

### 3. 信号绿的角色

绿色不是主背景色，而是系统信号色。

只允许用于：

- 状态点
- 激活条
- 协议标签高亮
- 扫描线
- 节点
- hover 激活描边
- 关键数值强调

不允许用于：

- 大面积背景
- 主色按钮海洋
- 整屏绿黑终端感

### 4. 页面角色单一原则

每一屏只能有一个主场景身份：

- Banner 2 是浅色实验台
- Banner 3 是深色验证主场
- Banner 4 是浅色审计档案区

不能再出现“浅底想做背景，深色卡又抢主舞台”的双中心问题。

## 五、Banner 2：决策入口矩阵（Calibration Deck）

### 1. 角色定义

Banner 2 的角色不是测试列表区，而是：

浅色实验台 / 校准操作台

用户从深色主控舱出来，进入一个结构化、冷静、可操作的选择层。

用户在这里应该感受到：

- 不是“做个测试玩玩”
- 而是“我在选择合适的判断协议”

### 2. 这一屏要解决什么问题

当前第二屏的问题是：

- 变浅太快
- 太像内容区
- 与 Hero 断层明显
- 6 宫格虽有系统风，但整体仍像排版在白纸上

新方案目标：

- 让它成为从深色启动舱降维展开出来的模块工作台
- 既释放 Hero 的压迫感，又不丢失品牌系统感

### 3. 背景设计

背景角色：

冷雾实验台，而非白底内容区。

背景颜色主渐变：

- 上部：`#DDE4EA`
- 中部：`#E6EBF0`
- 下部：`#EDF1F4`

材质表现必须叠加：

- 极弱密度的细网格
- 极浅金属雾面纹理
- 低对比纵向校准线
- 极淡的实验台边界线

顶部与 Banner 1 的衔接方式：

- 取消波浪切边
- 改为能量衰减式过渡
- Hero 最下方保留 `80–120px` 高度的深色余辉带
- 余辉向下渐隐成浅雾灰
- 在过渡区放 `3–5` 根极细垂直线
- 垂直线带有微弱绿色残影，模拟系统扫描束继续向下传导

这比白色波浪切边更符合工业系统品牌。

### 4. 布局结构

从上到下：

1. 小号协议标签
2. 主标题
3. 一句解释性正文
4. 中央主矩阵区
5. 底部轻量辅助说明 / 分类入口

宽度：

- 内容容器：`1200–1280px`
- 核心矩阵主容器：`880–1040px`
- 不要过宽，避免像表格铺满页面

纵向留白：

- 顶部留白：`96–120px`
- 标题到矩阵：`40–56px`
- 矩阵到底部说明：`32–48px`
- 整屏总高度：`820–980px`

### 5. 文案建议

顶部协议标签：

`DECISION ENTRY LAYER / PROTOCOL READY`

主标题：

`决策入口矩阵`

副文案：

不是选择一个“好玩的测试”，而是进入与你当前判断任务相匹配的测量协议。

这句必须保留，因为它直接把首页从娱乐测试站拉开。

### 6. 核心矩阵设计

结构形态：

6 个入口不是普通卡片，而是嵌入实验台中的协议槽位 / 模块抽屉。

排列方式：

- 桌面端 `3 × 2`
- 每个模块宽高比约 `1.15 : 1`
- 模块之间留 `0` 间距或极细分隔线
- 整体更像一个展开的系统机柜，而不是 6 张漂浮卡片

模块内容层级：

1. 协议标签
2. 模块名
3. 解释文案
4. 适用任务标签
5. 状态条 / 激活值
6. 进入动作

模块文案结构示意：

上方：

`TYPE: ROLE-FIT / SLOT: R2-C1`

标题：

`MBTI 性格测试`

说明：

用于识别信息获取、判断与决策偏好，适合在学业方向、合作风格与角色定位中建立基础认知。

标签：

`学业选择 / 沟通风格 / 决策方式`

右下：

`ACTIVE`

### 7. 模块视觉细节

卡槽材质：

- 底色比背景稍深一层冷灰
- 表面像细雾面金属板
- 不要大投影
- 不要明显悬浮感

边框：

- 常态：`1px solid rgba(16,22,31,0.08)`
- hover：边框局部转为信号绿
- hover 时仅在一侧或顶边出现激活线，不要全框强绿

圆角：

- `4px` 或 `6px`
- 趋近直角
- 不要 `16px` 以上大圆角

内部网格：

- 每个模块可带极弱网格 / 刻度线
- 强度必须很低，避免抢字

激活反馈：

hover 时发生三件事：

- 顶部协议标签轻微提亮
- 一条绿色状态线从左向右扫过 `20–30%`
- 标题字重或亮度微升

不要出现：

- 大面积发光
- 弹跳
- 卡片浮起很多
- SaaS 风 hover 大阴影

### 8. 底部辅助区

矩阵底部不能直接空掉。

左侧建议：

`RACK_STATUS: ACTIVE / GRID: 3 × 2 / MODULES: 6`

右侧建议：

一个轻量筛选入口：

- 协作与角色
- 职业与认知
- 压力与情绪
- 能力与表现

它不是功能重点，但能增强实验台操作层的感觉。

### 9. Banner 2 动效建议

进入动画：

- section 进入时背景雾层先出现
- 模块边线从低对比逐渐显现
- 标题先出，矩阵延迟 `120ms` 再整体出现

hover 动效：

- 用横向扫描线
- 用局部边框亮起
- 不做飘浮卡片动效

滚动氛围：

- 网格背景可有极弱视差
- 但不能明显像炫技 landing page

### 10. Banner 2 移动端原则

- 改为 1 列堆叠
- 每个模块像系统清单项
- 顶部标签仍保留
- 浅色实验台背景不能丢
- 不要变成普通白底卡片流
- 模块高度控制在 `180–220px`
- 只保留最关键的信息层级

## 六、Banner 3：三大架构协议（Core Engine / Validation Rack）

### 1. 角色定义

这一屏必须明确为：

深色验证机架 / 计算内核区

它承担的不是介绍，而是证明。

用户看到这一屏时必须意识到：

- 费马测试不是标签拼图游戏
- 它有底层测量架构
- 有常模
- 有规则映射
- 有从特质到现实决策的转换逻辑

### 2. 为什么这一屏必须重新回到深色

因为这一屏代表进入系统内部。

Banner 2 是操作台。  
Banner 3 应该是机房 / 核心舱 / 深层协议区。

如果这里继续是浅底，会削弱品牌的内核感。

因此建议：

- Banner 3 整体做成完整深色段
- 不要再用浅底包裹深色卡

### 3. 背景设计

背景色：

- 顶部：`#0A0F14`
- 中部：`#10161F`
- 下部：`#151D27`

材质：

- 极低频噪点
- 微弱网格
- 局部暗色雾影
- 很淡的纵向显像管条纹感

与 Banner 2 的衔接：

- 不要波浪
- 不要花哨渐变
- 建议使用硬切 + 协议警戒线
- Banner 2 结束处是一条极细水平线
- 进入 Banner 3 顶部时出现一条绿色系统分隔线
- 旁边带一条超小协议字：

`[ ENTERING CORE PROTOCOL ZONE ]`

这比品牌装饰更像工业系统。

### 4. 布局结构

从上到下：

1. 协议标签
2. 章节标题
3. 解释性短句
4. 核心三栏协议机架
5. 下方一条简短总结语

容器宽度：

- 外层容器：`1200–1280px`
- 核心机架：`960–1080px`

纵向节奏：

- 顶部留白：`120–144px`
- 标题下方：`20–24px`
- 说明文与机架：`40–56px`
- 机架下总结：`28–40px`
- 整体高度：`780–920px`

### 5. 文案建议

协议标签：

`THE FERMAT PROTOCOLS / CORE ARCHITECTURE`

主标题：

`三大架构协议`

副文案：

这不是三套不同的卖点，而是同一套测量系统得以成立的底层结构。

### 6. 三栏机架结构

这里不是普通 3 张介绍卡，而是一台被横向切开的三 Bay 服务器机架。

每一栏对应一个核心协议支柱：

1. 高分辨率测量
2. 常模锚定系统
3. 场景映射引擎

推荐比例：

- 三栏等宽
- 中间竖线分隔
- 外框完整闭合
- 顶部有统一协议头

整体必须像一块完整设备，而不是三张卡拼一起。

### 7. 三个模块的内容设计

#### 模块 A：高分辨率测量

标题：

`高分辨率测量`

标签：

`FACETS: 30 / SCAN_DEPTH: HIGH`

核心图形：

- 不要图标
- 使用一个 `30-Facet` 点阵矩阵
- 推荐 `5 × 6` 或更接近真实矩阵逻辑的布局
- 部分点亮为绿色
- 部分点为冷灰空心
- 点与点之间有很淡的联接关系

说明文：

以 `30-Facet` 分面矩阵代替粗颗粒度人格标签，将认知扫描推进到更高分辨率。

底部系统字：

`SAMPLE_RATE: HIGH / TRACEABLE_STATE: ENABLED`

#### 模块 B：常模锚定系统

标题：

`常模锚定系统`

标签：

`NORM_SET: 100K+ / Z-SCORE: ACTIVE`

核心图形：

- 一个高斯曲线 + 动态游标
- 中央曲线是暗绿色线
- 游标位置显示当前用户在常模中的相对坐标
- 下方刻度为 `-2σ / -1σ / M / +1σ / +2σ`

说明文：

个体分数不被孤立解释，而是被放置在常模系统中进行动态定位与偏差校准。

底部系统字：

`NORM_REFERENCED / STABILITY_CHECK: PASS`

#### 模块 C：场景映射引擎

标题：

`场景映射引擎`

标签：

`SCENARIO_ROUTER / ACTIONABLE_OUTPUTS`

核心图形：

- 一个节点拓扑图
- 左侧是特质节点
- 中间是规则映射层
- 右侧是输出场景槽位
- 用细虚线连接
- 某几条路径亮绿激活

说明文：

系统不是停留在特质描述，而是将结构结果映射进真实任务与判断情境。

底部系统字：

`RULE_LOCK: 99.9 / OUTPUT_MODE: ACTIONABLE`

### 8. 视觉细节

外框：

- 深色机架整体用完整边框围合
- 边框颜色：`rgba(242,242,247,0.14)`

分隔线：

- 竖向分隔线略亮于背景
- 不是高对比白线
- 像设备仓体之间的结构线

标题颜色：

- 主标题白度高一些
- 协议标签低对比
- 说明文压低亮度

内部组件感：

- 不要大面积高亮块
- 用线、点、刻度、游标建立可信感

## 十二、4 个需要补足的硬核落地维度

### 1. 业务状态层的缺失：Banner 2 的断点续录 UI

这里必须从产品系统角度补足。

仓库事实：

- 当前答题引擎已具备本地状态缓存能力
- Big5 状态通过 `zustand + persist + localStorage` 持久化
- 已持久化字段包括 `attemptId`、`resumeToken`、`answers`、`currentIndex`、`lastSubmittedAt`
- 存储 key 为 `fm_big5_attempt_v1`

这意味着 Banner 2 不能假设所有用户都是首次进入。

#### 补充设计

在 Banner 2 的 `6` 宫格矩阵中，需要设计“任务进行中（In-Progress）”状态。

UI 表现：

- 常态：`PROTOCOL_READY`
- 当检测到缓存中有未完成答题进度时，右上角标签变为 `[ CACHE_FOUND: 45% ]`
- 伴随一个缓慢闪烁的琥珀色 warning 点或信号绿圆点
- hover 时动作不再是 `[ INITIALIZE ]`
- 而是 `[ RESUME_SCAN ]`

这能显著增强“这是一套带存档功能的工业系统”的真实感。

### 2. 支付与鉴权边界的物理化（The Paywall Threshold）

作为接入商户支付的 SaaS 引擎，用户何时、以什么姿态遇到权限墙，必须被设计进首页中段。

仓库事实：

- 已存在支付等待页与订单恢复链路：[pay/wait/page.tsx](/Users/rainie/Desktop/GitHub/fap-web/app/(localized)/[locale]/pay/wait/page.tsx#L1)
- E2E 已覆盖 `wechatpay`、`alipay`、wait flow、payment recovery token 等支付语义：[clinical-combo-flow.spec.ts](/Users/rainie/Desktop/GitHub/fap-web/tests/e2e/clinical-combo-flow.spec.ts#L354)
- 当前 `OfferCard` 更像通用白底电商卡片，不适合代表“鉴权终端”

#### 补充设计

在 Banner 2 或 Banner 3 中，必须通过 UI 明确划分：

- 公共协议
- 高级授权协议

UI 表现：

- 需要高级权限的模块，在槽位右上角增加极小的 `[ ENCRYPTED ]` 或 `[ REQ_AUTH ]`
- 旁边带一个 `1px` 结构化锁定图标
- 不要用金币、皇冠、钻石等消费化符号

当点击触发支付请求时：

- 不要弹出传统扫码付款大圆角弹窗
- 而应该出现居中的“鉴权终端（Authorization Terminal）”
- 提示文案建议为 `[ INITIATING_SECURE_PAYMENT_GATEWAY ]`

这样商业化触点才不会破坏整个系统的冷峻感。

### 3. 前端实现级的视觉翻译（Tailwind / Motion 指南）

像“能量衰减”“极低频噪点”“显像管条纹”这类词必须被翻译成工程约束，否则前端很容易做成廉价渐变或高成本图片。

#### 补充设计：能量衰减（断点 A 修复）

不建议用普通 CSS 线性渐变直接处理。

建议方式：

- 用一层伪元素承接 Hero 底部
- 通过 `mask-image: linear-gradient(...)` 控制吃色
- 叠加极低透明度深色雾影
- 必要时用 `mix-blend-mode: overlay`

目标是把深色系统逐步稀释掉，而不是做一个普通漂亮渐变。

#### 补充设计：显像管条纹与噪点（Banner 3）

坚决不用图片。

建议方式：

- 使用 SVG Filter 生成静态 `feTurbulence` 噪点
- 通过 Tailwind 或 CSS 设为背景层
- 透明度控制在 `0.02 - 0.04`

这样可以保证：

- 页面轻量
- 任意分辨率不失真
- 不产生高带宽纹理资源

#### 补充设计：动态扫描线

你原要求是用 `Framer Motion` 实现：

- 一条 `1px` 高的 `#00FF41` div
- 从 `y: "-100%"` 到 `y: "100%"` 无限循环
- `ease: "linear"`、`duration: 3`
- 叠加 `box-shadow: 0 0 8px #00FF41`

复核结论：

- 这个动效定义是合理的
- 但当前仓库 `package.json` 还没有 `framer-motion`

因此执行时有两条路径：

- 如果允许新增依赖，按你这套 `Framer Motion` 规范做
- 如果当前阶段不想扩依赖，先用 CSS keyframes 或 WAAPI 等价实现

文档层面必须保留这条要求，不能省略。

### 4. 移动端的终端机化（Terminal Adaptation）

“一列堆叠”是不够的。移动端必须拥有明确形态。

#### 补充设计

当宽度 `< 768px` 时，Banner 2 和 Banner 3 应表现得像一台手持诊断仪（Handheld Scanner）。

UI 表现：

- 取消左右大留白
- 模块边框尽量顶到屏幕边缘，可近似 `w-full + border-x-0`
- 模块之间的分割线加粗，例如 `border-b-2`
- 形成类似军用设备的坚固感
- 核心数据图表在移动端不要简单缩小
- 更合理的方式是截取局部并允许 `overflow-hidden`
- 让用户感受到图表的庞大，而不是看到一个被压扁的小缩略图

这条必须写进移动端规范，因为它决定移动端是否仍像同一台系统设备。
