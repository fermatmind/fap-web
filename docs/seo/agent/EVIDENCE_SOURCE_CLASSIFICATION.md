# Evidence Source Classification

Status: MVP0 contract

Every Control Packet evidence source must be classified before it can influence a recommendation.

## Classification Matrix

| Source class | Definition | Allowed use | Forbidden use | GPT 5.5 Pro content recommendations | Opportunity scoring | CMS draft support |
|---|---|---|---|---|---|---|
| `LIVE_API` | Current read-only backend, CMS, `seo_intel`, public runtime, or provider export evidence. | Operational truth when access and payload are verified. | Mutation without approval; replacing CMS authority with frontend fallback. | Yes, if claim and source limits are preserved. | Yes, if freshness and joins pass. | Yes, for dry-run package planning only. |
| `STATIC_ARTIFACT` | Repo artifact checked in or attached as a static sample. | Contract shape, dashboard shell, fixture comparison, planning. | Production truth, live traffic conclusions, CMS mutation. | Limited; must be labeled as non-live. | No. | No, except as package-shape reference. |
| `GENERATED_ARTIFACT` | Report or artifact generated by a prior run. | Evidence index, audit trail, prior hypothesis. | Final truth after later state changes; direct production action. | Limited; must confirm with current authority before recommendations. | No unless regenerated from verified live sources in the same packet. | No unless paired with current CMS/runtime evidence. |
| `MOCK` | Hard-coded demo or placeholder data. | UI shell review, contract prototyping, test fixture expectations. | Production decisions, opportunity scoring, CMS/search action. | No, except to comment on UI/contract shape. | No. | No. |
| `FIXTURE` | Deterministic test fixture or sample collector output. | Tests, schema validation, parser behavior, local examples. | Live performance conclusions, opportunity scoring, provider submission. | No, except to evaluate format. | No. | No. |
| `UNKNOWN` | Source exists but freshness, authority, or live status is not verified. | Mark access gap, request more evidence, block risky actions. | Production decisions or mutation. | Only to identify missing evidence. | No. | No. |
| `ACCESS_REQUIRED` | Source requires login, export, operator screenshot, private API, or production read access not available in the run. | Mark as inaccessible and request operator evidence. | Guessing values or treating absence as negative proof. | Only to ask for evidence. | No. | No. |
| `MIXED` | Composite source with more than one class, such as mock KPI plus static issue artifact. | Review only after component classes are listed. | Collapsing mixed data into production truth. | Only on the verified components. | Only if every scoring component is live and quality-checked. | Only if CMS-relevant components are live and approved. |

## Required Evidence Labels

Use these labels on every material finding:

- `VERIFIED`: current evidence supports the statement.
- `NOT_VERIFIED`: checked but not proven.
- `ACCESS_REQUIRED`: evidence is behind a private surface or export.
- `NEEDS_OPERATOR_CONFIRMATION`: evidence suggests an issue but operator/CMS policy must confirm.
- `UNKNOWN`: no reliable evidence yet.

## Examples

| Evidence | Source class | Evidence label | Decision impact |
|---|---|---|---|
| Public runtime `/en/articles/mbti-basics` 308 redirect | `LIVE_API` | `VERIFIED` | Runtime QA can report; CMS route intent needs confirmation. |
| `/ops/seo-operations` traffic trend mock array | `MOCK` | `VERIFIED` | Cannot enter opportunity scoring. |
| `docs/seo/generated/seo-issue-queue.v1.json` | `STATIC_ARTIFACT` | `VERIFIED` | Can verify dashboard shell only. |
| `GscCollector` fixture output | `FIXTURE` | `VERIFIED` | Blocks live GSC conclusions. |
| Production `seo_gsc_daily` row freshness | `UNKNOWN` | `NEEDS_OPERATOR_CONFIRMATION` | Requires GSC Data Quality PR. |

## Promotion Rules

Evidence may be promoted only by adding a stronger source, not by rewording the packet.

- `MOCK` cannot become `LIVE_API`.
- `FIXTURE` cannot become live data.
- `UNKNOWN` becomes `LIVE_API` only after a verified read-only source proves freshness and authority.
- `ACCESS_REQUIRED` becomes `VERIFIED` only after the operator provides export/screenshot/API evidence.
- `MIXED` remains mixed until every component is individually classified.
